# EVE legacy font metric block declared L2 (17) instead of L4 (2) — masked by serial-only bench verification, caught by code review

## Problem

The dash renders all text through 8 custom EVE bitmap-font instances
(`_Static_assert(DASH_FONT_COUNT == 8, ...)`, `tests/test_dash_fonts.c:22`),
each a BT81x "legacy font": a 148-byte metric block plus L4 (4-bit grayscale)
glyph cells, generated by `tools/make_dash_fonts.py` into
`MustangDash/dash_fonts.h` and registered on the BT817 via `CMD_SETFONT2`
(`MustangDash/MustangDash.ino:488-490`).

The metric block's format field at offset 128 tells the chip how to decode the
glyph data. Pre-fix, the generator wrote **17** into that field. In the
vendored library's authoritative table (`libraries/FT800-FT813/src/EVE.h`),
17 is not L4:

- `EVE.h:190` — `#define EVE_L4 ((uint8_t) 2)`
- `EVE.h:261` — `#define EVE_L2 ((uint32_t) 17UL)`

So every glyph — data packed 2 pixels per byte as L4 — was declared to the
renderer as L2 (4 pixels per byte, 2-bit). The firmware booted, held 60 fps,
and acknowledged every serial command; only the pixels were wrong. Fixed in
PR #3 (branch `feat/dash-layout`, open as of this writing).

## Symptoms

There were none observable through the verification channel in use — that is
the symptom story:

- Boot banner normal; `EVE_init()` returned `E_OK` as always.
- All 8 fonts uploaded and passed the firmware's own integrity check — the
  loader inflates each glyph stream and verifies `CMD_GETPTR` landed exactly
  at `gaddr + f->glyph_bytes` (`MustangDash/MustangDash.ino:440-445`). Byte
  counts are format-agnostic, so this check passes regardless of the format
  field.
- The host invariant suite was green — the pre-fix test asserted the format
  field **was** 17.
- Frame rate and RAM_G totals normal (`MustangDash/MustangDash.ino:468-469`
  prints the font footprint and headroom; it matched the plan).

The defect was only visible by looking at the panel — which nobody had done.
"Hardware-verified" at that point rested entirely on serial evidence.

## What Didn't Work

1. **The host invariant test.** The pre-fix assertion pinned `format field
   must be 17 (L4)` — it encoded the converter's own misunderstanding, not an
   authoritative source. A test that mirrors the implementation's belief
   cannot catch the belief being wrong; it converts a bug into an invariant.
2. **Serial-channel bench verification.** fps counters, RAM_G byte totals,
   the `CMD_GETPTR` inflate checks (`MustangDash/MustangDash.ino:444`), and
   command acks all pass no matter what the pixels look like. The channel
   could not carry the failure.
3. **The library's type system.** The format field is a bare little-endian
   u32 inside a byte array (`tools/make_dash_fonts.py:136` packs it with
   `struct.pack("<5I", EVE_L4, stride, cell_w, cell_h, 0)`); 17 is a
   perfectly valid — wrong — value. Nothing faults, nothing warns.

## Solution

The code-review correctness reviewer caught it by quoting `EVE.h` against the
converter, and cross-checked Bridgetek's own converted L4 font binaries, which
store `02 00 00 00` at offset 128. Fix flow (PR #3):

1. **Constant corrected in the generator.** `tools/make_dash_fonts.py:43-46`
   now reads:

   ```python
   # BITMAP_LAYOUT format code for 4-bit grayscale. The BT81x format table
   # (and the vendored library's EVE.h:190) define L4 = 2; 17 is L2. A wrong
   # value here renders every glyph through the wrong bit depth.
   EVE_L4 = 2
   ```

2. **Headers regenerated deterministically** (`wsl -- python3
   tools/make_dash_fonts.py`; the script's contract is byte-identical output
   for a given Pillow/FreeType, `tools/make_dash_fonts.py:27-29`), so
   `MustangDash/dash_fonts.h` now carries 2 at offset 128 of every metric
   block.

3. **Test re-pinned with the source cited.** `tests/test_dash_fonts.c:157`:

   ```c
   expectf(rd32(m, 128) == 2U, name, "format field must be 2 (EVE_L4 per EVE.h:190; 17 would be L2)");
   ```

   and the file header (`tests/test_dash_fonts.c:4-5`) states the rule with
   its source: "L4 format code 2 (EVE.h EVE_L4; 17 is L2)".

4. **Reflashed and visually verified.** Kevin eyeballed the panel
   (2026-07-10): the pre-fix glyph corruption had been visually real —
   his words, "the font fix fixed the biggest issue" — and the corrected
   fonts render properly. The eyeball came a step after the fix landed
   (the doc briefly and honestly carried an "eyeball still owed" note),
   which is itself the bug's lesson operating: the claim waited for the
   right verification channel instead of borrowing a green serial log.

Verified in the current tree (host side): the test compiles and passes under
WSL (`OK: dash fonts honor the EVE legacy format (148B metrics, L4, even
cells, widths, zlib; 273120 B decoded)`).

## Why This Works

The legacy 148-byte metric block is what `CMD_SETFONT2` registration derives
the handle's `BITMAP_LAYOUT` from: 128 advance-width bytes, then u32 LE
format / stride / cell width / cell height / glyph pointer (packed at
`tools/make_dash_fonts.py:136`; consumed at
`MustangDash/MustangDash.ino:449-455`, where the firmware patches gptr, and
`:488-490`, where the block address is handed to `EVE_cmd_setfont2`). The
format field at offset 128 is therefore the single source of the decode bit
depth. Declaring L2 makes the renderer read 4 pixels per byte from data
packed 2 pixels per byte — half of every cell's data interpreted at the wrong
width, every frame, with no error path.

The authoritative mapping lives in the library header, not in anyone's memory
of a documentation table: `libraries/FT800-FT813/src/EVE.h:189-191` (`EVE_L1`
= 1, `EVE_L4` = 2, `EVE_L8` = 3) and `EVE.h:261` (`EVE_L2` = 17). The test
now asserts against that spec value *with the source named in the failure
message*, so any regression tells the reader where truth lives instead of
re-litigating which number is right.

## Prevention

1. **Pin tests to authoritative constants, not implementation beliefs.**
   When a test asserts a magic number, the assertion (or its failure message)
   must cite where the number comes from — a library header, a spec section.
   The current form is the model (`tests/test_dash_fonts.c:157`):

   ```c
   expectf(rd32(m, 128) == 2U, name, "format field must be 2 (EVE_L4 per EVE.h:190; 17 would be L2)");
   ```

   A test written by reading the implementation it tests is a mirror, not a
   check — the pre-fix suite proved that by staying green around the bug.

2. **Match the verification channel to the failure mode.** Serial evidence —
   counters, acks, byte totals, `CMD_GETPTR` — cannot validate rendering.
   Bench rule: **rendered output changed → someone looks at the panel before
   "hardware-verified" is claimed.** If eyes-on isn't possible, a capture
   (photo, or a RAM_G/snapshot readback) substitutes; a green serial log does
   not.

3. **Confusable-constant hygiene.** When writing format/mode codes into
   binary structures, reference the library constant symbolically in the
   generator, or assert the generator's constants against the library header
   in a test — never re-derive numeric literals from documentation tables.
   The EVE format table is a trap by construction: L1/L4/L8 are 1/2/3 but L2
   is 17 (`EVE.h:190` vs `EVE.h:261`), so "L*n* ≈ small consecutive numbers"
   intuition produces exactly this bug. The generator now carries the
   cross-reference and the warning inline (`tools/make_dash_fonts.py:43-46`).
